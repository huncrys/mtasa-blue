# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

This is a fork of [multitheftauto/mtasa-blue](https://github.com/multitheftauto/mtasa-blue/) — Multi Theft Auto: San Andreas, a multiplayer mod for GTA:SA. This fork's key difference from upstream: it uses **LuaJIT 2.1** instead of stock Lua 5.1 (`vendor/luajit`). CI/releases are hosted on a self-hosted GitLab instance (`oaklab.hu`), not GitHub.

The repo currently tracks upstream's `release/1.6.0` branch. Remotes: `origin` (`git@oaklab.hu:crys/mtasa-blue.git`, this fork) and `upstream` (`git@github.com:multitheftauto/mtasa-blue.git`). When pulling in upstream changes, diff/merge against `upstream/release/1.6.0`.

The codebase is a large C++17 monorepo split into:

- `Client/` — the game client mod (only buildable on Windows; injects into/wraps GTA:SA). Includes `core` (main client engine), `game_sa` (GTA:SA game interface layer), `gui`, `loader`, `multiplayer_sa`, and `mods/deathmatch` (the actual gamemode logic, analogous to Server's `mods/deathmatch`).
- `Server/` — the dedicated server (`core`, `mods/deathmatch`, `dbconmy` for MySQL, `launcher`). Builds on Windows, Linux, and macOS.
- `Shared/` — code compiled into both client and server: `sdk/` (core utility types — `SharedUtil.*`, `CVector`, `SString`, networking RPC enum definitions in `sdk/net`), `mods/deathmatch` (shared gamemode logic), `XML`, `animation`, `publicsdk` (public headers consumed by module/resource authors).
- `vendor/` — third-party dependencies vendored directly into the tree (not git submodules), each with its own `premake5.lua`: LuaJIT, CEF3, mbedtls, cryptopp, zlib, sqlite, pcre, discord-rpc, etc.
- `utils/` — build tooling: premake5 binaries, `buildactions/` (premake Lua scripts that install CEF/resources/data/fonts as part of the build), localization tooling, clang-format helpers, docker build images.

### Client/Server symmetry

Most gameplay logic exists in parallel on both sides under `mods/deathmatch/logic`, communicating over RPC. Networked entity actions flow through `CElementRPCPacket` (Server) and per-feature RPC handlers under `Client/mods/deathmatch/logic/rpc/*RPCs.cpp` (e.g. `CWeaponRPCs`, `CMarkerRPCs`, `CPlayerRPCs`). RPC packet IDs are defined in `Shared/sdk/net/rpc_enums.h`.

### Lua scripting API

Lua natives (the scripting functions resource authors call, e.g. `getPlayerName`) are implemented in `CLuaFunctionDefs*.cpp` under `Client/mods/deathmatch/logic/lua/` (client) and the server equivalent. Functions are registered through `CLuaCFunctions`/`CLuaManager` and grouped by feature area (`.BodyClothes`, `.Commands`, `.Event`, `.Input`, `.Notification`, `.Output`, `.Voice`). When adding or changing a scripting function, the definition, its registration, and (for shared behavior) any client/server RPC plumbing all need to stay in sync.

## Build commands

Builds are driven by [premake5](https://premake.github.io/) (`premake5.lua` at the repo root, with per-module `premake5.lua` files throughout `Client/`, `Server/`, `Shared/`, `vendor/`).

### Linux / macOS (server only — client is Windows-only)

```bash
./linux-build.sh --arch=x64 --config=release   # arch: x86|64/x64|arm|arm64, config: debug|release
```

This regenerates `Build/` and `Bin/` from scratch (`rm -Rf Build/ Bin/`), runs premake to produce GNU makefiles, then `make`. Env vars `GCC_VERSION`, `CC`/`CXX`/`AR`, `NUM_CORES` (defaults to nproc), `LUAJIT_CC` can override toolchain selection; `GCC_PREFIX` is used for cross-compiling arm/arm64. Server output lands in `Server/output/`.

To regenerate build files only (without compiling), invoke premake directly:

```bash
utils/premake5 --os=linux --file=premake5.lua gmake
```

### Windows

```bat
win-create-projects.bat   REM generates Build/MTASA.sln via premake5 (vs2022 action)
win-build.bat [Debug|Release] [Win32|x64|ARM|ARM64]   REM locates MSBuild via vswhere and builds the solution
```

### CI

`.gitlab-ci.yml` + `.steps.gitlab-ci.yml` define the GitLab CI pipeline (prepare → build → package → upload → release), parameterized per platform (x86/x64; arm/arm64 currently disabled). CI builds set `Shared/build_overrides.h` to pin `MTASA_VERSION_BUILD` to the network module's revision before building — this file is generated by CI and should not normally be hand-edited outside of testing a specific version pin.

There is no unit test suite for the C++ codebase; `utils/localization/tests` covers only the localization tooling.

## Conventions

- C++17, 4-space indentation for `.h/.hpp/.c/.cpp/.cc` (`.editorconfig`), formatted via `.clang-format` (Google-based, 4-space indent, 160 col limit — run `clang-format` before committing changes to those files).
- Shell scripts use LF line endings.
